# Reproducibility Test Data

Test fixtures for the current Agora reproducibility flow using the CSV comparison scorer in `containers/repro-scorer`.

This folder is meant to feel like a small but believable computational reproducibility bounty:
- public input data is given to solvers
- a reference output is used for scoring
- solvers submit a reproduced CSV artifact
- score reflects row-by-row agreement with optional numeric tolerance

Important realism note:
- this is an **open reference benchmark**, not a blind holdout evaluation flow
- the reference output is part of the posted challenge materials in the current protocol
- use this template for public artifact-matching and benchmark tasks, not for secret-answer scoring

## Current Runtime Contract

The reproducibility scorer currently expects:
- `ground_truth.csv`: the reference output artifact
- `submission.csv`: the solver's reproduced output
- identical column names between both files

The built-in preset is:
- `CSV Comparison`
- official container: `ghcr.io/agora-science/repro-scorer:latest`

## Files

### Challenge posting inputs

| File | Use in UI | Purpose |
|------|-----------|---------|
| `input_dataset.csv` | Source dataset | Public source data solvers work from |
| `expected_output.csv` | Reference output | Public reference artifact used for scoring |

### Solver submission fixtures

| File | Expected result | Purpose |
|------|-----------------|---------|
| `sample_submission.csv` | Valid, perfect score | Exact reproduction of expected output |
| `near_match_submission.csv` | Usually valid if tolerance allows | Small numeric drift within tolerance |
| `bad_submission_missing_column.csv` | Invalid | Missing a required output column |
| `bad_submission_extra_rows.csv` | Valid but penalized | Reproduced artifact has mismatched row count |

### Local scorer convenience

| File | Purpose |
|------|---------|
| `ground_truth.csv` | Direct local alias for `expected_output.csv` |

## Recommended Human UI Test Plan

## 1. Post an open reproducibility benchmark

Use these values in `/post`:

### Step 1: Scientific Brief
- Bounty title: `Reproduce normalized assay summary statistics from a published pipeline`
- Challenge brief:
  `We provide a public input dataset and expect solvers to reproduce the normalized summary table generated by the reference pipeline.`
  `Submissions should be CSV artifacts with the same columns as the expected output.`
  `Scores are computed by row-by-row comparison against the reference artifact.`
- Reference paper or protocol link (optional): add a paper, notebook, or methods page if you have one
- Marketplace category: `Longevity` or `Other`
- Keywords (optional): `reproducibility, csv, benchmark`

### Step 2: Data & Reference Artifacts
- Source dataset: `input_dataset.csv`
- Reference output: `expected_output.csv`
- Match rule: `Allow small drift`
- Allowed drift: `0.001`

### Step 3: Submission & Scoring
- Submission artifact: fixed to CSV output only
- Official scoring rule: deterministic CSV comparison
- Solver notes (optional):
  `Rows are compared positionally against the public reference output, with numeric values accepted within tolerance 0.001.`

### Step 4: Reward & Timeline
- Reward pool: `10`
- Payout rule: `Winner takes all`
- Submission window: `30 min` for testnet E2E, or `7 days` for a production-like posting walkthrough
- Review window before payout: `0` for fast testnet E2E, or any longer value you want to test operationally

## 2. Submit the exact reproduction

Use:
- `sample_submission.csv`

Expected current behavior:
- perfect score
- all rows matched
- no row-count penalty

## 3. Submit the near-match reproduction

Use:
- `near_match_submission.csv`

Why:
- validates the tolerance field in a realistic way
- useful for checking whether the challenge poster chose a tolerance that is too strict or too loose

Expected current behavior:
- with tolerance `0.001`, this should still score very highly or perfectly
- with tighter tolerance, some rows may stop matching

## 4. Submit malformed or penalized outputs

### Missing column
Use:
- `bad_submission_missing_column.csv`

Expected current behavior:
- rejected because the submission is missing a required column from the expected artifact

### Extra rows
Use:
- `bad_submission_extra_rows.csv`

Expected current behavior:
- accepted but penalized for mismatched row count
- useful for understanding how the current scorer handles over-produced output artifacts

## Why This Folder Is Realistic

A lot of reproducibility bounties are not arbitrary notebooks. They are effectively:
- `here is the input data`
- `here is the reference artifact`
- `reproduce this exact result within a documented tolerance`

That is exactly what this folder is modeling.

## What Each Posting Field Means In Practice

| UI field | Recommended value | Why |
|----------|-------------------|-----|
| Source dataset | `input_dataset.csv` | Gives solvers the public inputs they must reproduce from |
| Reference output | `expected_output.csv` | Defines the public benchmark artifact used for deterministic scoring |
| Match rule | `Allow small drift` | Reflects the current reproducibility scoring model when minor numeric noise is acceptable |
| Allowed drift | `0.001` | Matches the built-in scorer's absolute tolerance configuration |
| Submission artifact | `CSV output only` | Reflects the current fixed submission contract for reproducibility |
| Official scoring rule | `Deterministic CSV comparison` | Matches the official preset and current scorer implementation |
| Payout rule | `Winner takes all` | Best fit for a small reproducibility bounty |
